## 一、「组织即文件」方法论：一个把"防篡改 + 防注入"做进任务文件的工程案例

技能雷达本期新入看板的 **OthmanAdi/planning-with-files** 是「组织即文件」这条主张在任务粒度上的一个具体工程实现。它用三个 Markdown 文件把任务状态持久化在项目目录内——`task_plan.md`（路线图/阶段/决策）、`findings.md`（研究发现）、`progress.md`（完成度追踪），核心设计原则是"文件系统即记忆：状态存在文件里，不存在上下文里"，使 agent 在上下文压缩、清除甚至崩溃后仍能凭文件续接任务。

两个细节值得记录：

- **防篡改**：`/plan-attest` 命令可对 `task_plan.md` 做锁定校验，官方说明篡改后的计划文件在注入阶段会被拒绝。
- **防注入**：2.36.1 版本起在计划文件中加入 nonce 随机分隔符（`===BEGIN PLAN DATA===` 形式），防止被 prompt injection 篡改文件语义后误导 agent。

该项目 MIT 协议，经 GitHub API 核实（2026-07-14）**25,289 星、最后一次推送 2026-07-13**，仍在积极维护；支持 Claude Code、Cursor、Codex CLI、Kiro、OpenCode 等 60+ 平台，并提供简体/繁体中文变体（`/plan-zh`、`/plan-zht`）。跨引擎可用这一点，客观上也是「引擎随租随换、组织记忆不锁定单一平台」的一个外部印证样本。

来源：[GitHub - OthmanAdi/planning-with-files](https://github.com/OthmanAdi/planning-with-files)（README 与仓库元数据，一手）

## 二、人工闸门的一个样本：发送前弹出显式同意

同批入看板的 **AgriciDaniel/claude-obsidian**（基于 Karpathy LLM Wiki 模式的知识库 skill，GitHub 核实 9,329 星、MIT、最后推送 2026-05-28）本身是本地优先的纯 Markdown 知识图谱工具，核心操作不联网。它唯一涉及外传数据的路径——把 wiki 内容发给 Anthropic API 做检索增强——被设计成**发送前弹出明确同意提示**，且提供 `--no-llm` 参数可完全本地运行不触发该路径。

这个"对外动作前必须过一道人工确认"的具体实现，可以作为「人工闸门与对外守门」子领域的一个客观样本记录：闸门不必是抽象原则，也可以是一行"是否发送？"的确认提示。

来源：[GitHub - AgriciDaniel/claude-obsidian](https://github.com/AgriciDaniel/claude-obsidian)（一手）
